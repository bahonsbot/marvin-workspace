## Investigation — exec-completion leak patch lane [12:49 GMT+7]
- What I checked: traced the shared exec-completion leak through the active OpenClaw runtime after Philippe reproduced both the raw `System (untrusted)` lines and the verbatim internal `buildExecEventPrompt()` text in chat.
- Findings:
  - `server-node-events` converts `exec.finished` / `exec.denied` into untrusted queued system events and wakes heartbeat.
  - `heartbeat-runner` uses `buildExecEventPrompt()` for those runs, so `heartbeat-events-filter.ts` is part of the causal chain.
  - the likely shared visibility failure is later: `session-utils.fs` already recognizes the internal exec prompt text, but `isHiddenOpenClawTranscriptMessage()` only applies that internal-text check when the message role is `user`, which is probably too narrow.
  - a second gap is that raw `System (untrusted): [time] Exec finished ...` lines are not currently treated as hidden internal text.
- Output: wrote `projects/_ops/openclaw-exec-completion-leak-patch-proposal-2026-04-17.md` with the exact code path and a minimal patch recommendation.
- Recommended minimal fix: patch `session-utils.fs` so internal system-prompt text is hidden regardless of role, and extend its internal-text matcher to suppress raw exec system-event lines too.
- Why this lane: it is the smallest shared hide boundary and should affect both live transcript broadcast filtering and history hydration without removing internal exec context from reasoning.
- Rollback: remove the proposal note if it proves wrong; no runtime code changed in this investigation step.
