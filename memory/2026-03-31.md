## Mission Control chat reconnect investigation [12:44-13:03 GMT+8]

**Problem reported:** After sending a message in Mission Control Chat, Philippe was getting effectively kicked out of the live session. The message still sent, but the composer became unusable until a hard reload and reconnect.

### Root cause 1 — send-triggered bridge teardown
- File: `projects/mission-control/hooks/useRuntimeBridge.ts`
- Cause: the `load()` callback depended on `messages.length`.
- Effect chain:
  1. send appends user message
  2. `messages.length` changes
  3. `load()` identity changes
  4. `handleGatewayEvent()` identity changes
  5. websocket `useEffect(...)` dependency changes
  6. bridge effect tears down and reconnects after send
- Symptom match: exact fit for "message sends, then live chat disconnects / requires reload".
- Fix: moved hydration decision inside `setMessages((current) => ...)` and removed `messages.length` from `load()` dependencies.
- Commit: `d417389` — `mission-control: stop bridge reconnect on every sent message`

### Root cause 2 — summary-refresh session flicker
- Follow-up symptom after first fix: brief `SESSION CONNECTED -> SESSION WAITING -> SESSION CONNECTED` flicker, with composer still usable.
- Cause: websocket bridge effect still depended on broader runtime summary object paths, allowing summary refresh churn to recreate the bridge/session effect even when no material transport values changed.
- Fix: refactored bridge-effect inputs to stable scalar locals (`configured`, `browserReachability`, websocket URL, bridge token, gateway token) and narrowed the effect dependency list to those values.
- Commit: `1d4c3d2` — `mission-control: stabilize bridge effect deps on summary refresh`

### Deployment / verification
- Ran `npm run lint -- --file hooks/useRuntimeBridge.ts` after each patch: clean.
- Rebuilt and restarted Mission Control preview stack twice using standard helper scripts.
- Verified preview entry responded on `http://127.0.0.1:3005` after restart.
- Verified preview processes present: Next server, preview-origin proxy, runtime-bridge WS sidecar.

### User-observed result after deployment
- Hard kickout after send appears resolved.
- Secondary session wobble appears resolved or materially reduced.
- Philippe reported the chat now looks clean and that `SESSION CONNECTED` appears to return faster.

### Current status
- Treat as **resolved for now**.
- Reopen only if normal-use testing shows another reconnect wobble.
- If reopened, next audit layer should be WS sidecar / preview-origin proxy handshake, not first-pass React state.
