## Mission Control chat duplication regression hunt (Apr 2 late night)
- A new regression appeared in Mission Control Chat: user + assistant turns could duplicate a few seconds later, and in worse cases accumulate into many copies while the transcript sometimes jumped upward on its own.
- Initial hypotheses/patches focused on transcript sanitization and merge dedupe:
  - stripped Control UI wrapper/system metadata from hydrated history (`560bca0`, `ac7782d`)
  - loosened merge dedupe around hydrated/live assistant turns and main-session key normalization (`f272df0`, `50f7398`)
- Stronger clue from live behavior: duplicates could accumulate over time and the transcript sometimes scrolled upward by itself. That pointed to repeated active-session transcript rehydration, not just one-off wrapper replay.
- Current best fix applied: stop repeated transcript rehydration for the active chat session; hydrate on session change or when there is no meaningful transcript yet, then rely on live WS updates for ongoing conversation (`e162301`).
- Tactical lesson: for Mission Control Chat, repeatedly re-merging transcript history into an already-live active thread is high-risk and can cause both clone accumulation and scroll jumps. Prefer simpler hydration rules plus live bridge updates over clever signature-based rehydration for the active session.
