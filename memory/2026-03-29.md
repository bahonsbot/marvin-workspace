## OAuth Refresh Flow Discovery (afternoon session)

### Problem
Session fell back to `bailian/qwen3.5-plus` due to `refresh_token_reused` error on the `openai-codex:default` auth profile. This error occurs when OpenAI rejects a refresh token that has already been used to generate a new access token.

### Key Finding: Two Separate Auth Stores
- **Codex CLI OAuth**: stored in `~/.codex/auth.json` — refreshed via `codex login --device-auth`
- **OpenClaw Runtime OAuth**: stored in `~/.openclaw/agents/main/agent/auth-profiles.json` — refreshed via `openclaw models auth login`

These are independent; refreshing one does not update the other.

### Proper Runtime OAuth Refresh Command
```bash
openclaw models auth login --provider openai-codex --set-default
```

This runs the provider plugin's OAuth flow and applies the default model recommendation. The flow:
1. Generates an OAuth URL for browser authentication
2. User signs in at OpenAI in their local browser
3. User pastes the redirect/callback URL back into the terminal
4. Tokens are saved to `auth-profiles.json`

## Mission Control Transcript Rehydration Debug (evening)

### What was implemented
- Added a first server-side history path in `projects/mission-control/app/api/runtime-bridge/route.ts`.
- The route now accepts `sessionKey` and attempts to return `transcriptHistory` by reading the persisted OpenClaw session store under `/data/.openclaw/agents/main/sessions/`.
- Added client-side fetch wiring in `projects/mission-control/hooks/useRuntimeBridge.ts` so `/api/runtime-bridge` is requested with the active `sessionKey` and any returned `transcriptHistory.messages` seeds the live transcript state.
- Added a follow-up hydration trigger intended to re-load once the active target session key becomes known instead of relying only on the earliest initial load.

### What was verified
- Direct on-disk inspection earlier confirmed the needed data exists:
  - `sessions.json` maps `sessionKey` to the session file/id
  - per-session JSONL logs contain `message` rows with user/assistant content
- Mission Control rebuilt successfully after the first hydration patch and the preview route came back healthy.
- Philippe tested with a fresh message from `openclaw-control-ui` and confirmed the UI still came back with the old empty-state text:
  - `No live transcript yet...`

### Current status / likely failure zone
- The problem is no longer conceptual; it is in the implementation handoff.
- Most likely causes narrowed to:
  1. history request fires before the correct active session key is available
  2. session registry parsing in the route is not matching the actual `sessions.json` structure reliably
  3. the route returns history but the client later overwrites the seeded messages with empty state
- Important evidence from debugging:
  - API probing against `/api/runtime-bridge?sessionKey=...` was not cleanly verified yet
  - one registry probe hit a `KeyError: 'sessions'`, which suggests the route/parser assumptions about `sessions.json` structure need re-checking against the live file rather than relying on memory of the format
  - preview management also hit another `EADDRINUSE` / split-brain style issue during one restart attempt, reinforcing that preview/process state still needs disciplined handling via the runbook scripts

### Durable lesson
- Mission Control transcript rehydration needs a verified end-to-end trace, not just a code patch:
  - selected session key in UI
  - API request with that exact key
  - route lookup against live `sessions.json`
  - non-zero parsed history rows from the JSONL
  - seed applied without later client overwrite
- Until each link is verified with concrete counts, a plausible hydration patch can still fail silently and leave the old `No live transcript yet...` behavior in place.

### Resolved outcome
- Transcript rehydration is now working again in Mission Control Chat.
- The successful fix required three pieces together:
  1. route lookup against the real `sessions.json` shape (top-level sessionKey map, not `sessions.*`)
  2. client hydration merge instead of hard transcript replacement, so refreshes do not wipe live messages
  3. automatic hydrate trigger when `activeSessionKey` becomes known, matching the previously working manual Refresh path
- Hydrated transcript cleanup also now strips raw transport wrappers such as `[[reply_to_current]]`, untrusted sender metadata blocks, and timestamp envelope lines.
- Philippe verified the latest pass as successful: automatic reload hydration works and the metadata-wrapper noise is gone.
